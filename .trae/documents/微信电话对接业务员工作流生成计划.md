# 微信电话对接业务员工作流自动生成计划

## 背景

当前 `WorkflowView.vue` 中的 `createSandboxWorkflowForEmployee` 函数对所有员工生成相同的通用 3 节点工作流（start → employee → end），task 固定为 `analyze_document`，无法体现微信电话对接业务员的实际业务流程。

根据 `MOD_AUTHORING_GUIDE.md` 的描述，微信电话对接业务员的完整流程为：
> 副窗启用后 → Win32 监控微信来电 → 自动接听 → 采集 → ASR → 意图 → TTS → VB-Cable 回灌

沙盒预设 `phone_wechat` 已提供匹配的测试变量（`channel`, `topic`, `intent`, `call_state`）。

## 目标

当用户在沙盒实验室选择「微信电话对接业务员」后点击「一键生成该员工测试工作流」，自动生成一个 **7 节点 + 条件分支** 的完整业务流程工作流，可直接用于 Mock/真实测试。

## 生成的目标工作流图

```
开始
  ↓
监控微信来电 (employee: wechat_phone, task: monitor_incoming_call)
  ↓
来电状态判断 (condition)
  ├── call_state == 'ringing' → 自动接听 (employee: wechat_phone, task: auto_answer)
  │     ↓
  │   语音采集与ASR (employee: wechat_phone, task: asr_transcribe)
  │     ↓
  │   意图识别判断 (condition)
  │     ├── intent == 'answer' → TTS语音合成与回灌 (employee: wechat_phone, task: tts_playback)
  │     │     ↓
  │     │   结束
  │     └── (无条件兜底) → 结束
  └── (无条件兜底) → 结束
```

## 实现步骤

### 步骤 1：修改 `WorkflowView.vue` 的 `createSandboxWorkflowForEmployee` 函数

**文件**: `e:\成都修茈科技有限公司\MODstore_deploy\market\src\views\WorkflowView.vue`

在函数内，根据 `employeeId` 判断是否为微信电话对接业务员（匹配 `wechat_phone` 或含 `wechat_phone` 后缀的 id），如果是则生成上述 7 节点工作流，否则保持原有通用 3 节点逻辑。

核心改动逻辑：

```javascript
// 判断是否为微信电话对接业务员
const isWechatPhone = employeeId === 'wechat_phone'
  || employeeId.endsWith('-wechat_phone')
  || employeeId.endsWith('_wechat_phone')

if (isWechatPhone) {
  // 生成微信电话对接业务员专用工作流
  // 1. 创建工作流
  // 2. 创建 7 个节点（start, 3×employee, 2×condition, end）
  // 3. 创建 9 条边（含条件边）
  // 4. 自动选中 phone_wechat 预设
} else {
  // 保持原有通用逻辑
}
```

### 步骤 2：自动切换沙盒预设

当生成微信电话对接业务员工作流后，自动将 `sandboxPresetId` 设为 `phone_wechat`，并填充对应的运行变量 JSON，使用户无需手动选择预设即可直接测试。

### 步骤 3：验证

- 启动前端开发服务器
- 在沙盒实验室选择「微信电话对接业务员」
- 点击「一键生成该员工测试工作流」
- 确认生成了 7 节点 + 9 边的完整工作流
- 确认预设自动切换到 `phone_wechat`
- 点击运行，确认 Mock 测试通过
